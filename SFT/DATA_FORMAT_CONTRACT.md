# Контракт формата данных (SFT ↔ Data)

**Владелец:** SFT-трек · **Адресат:** Data-трек · **Статус:** v1 (MVP)

Этот документ фиксирует, в каком виде data-трек передаёт датасеты в SFT-трек.
Цель — подключать новый датасет **без переписывания тренировочного кода**.
Схема закладывается сразу под три задачи (drafting / polishing / editing),
но на MVP data-трек обязан отдать только **drafting**.

---

## 1. Физический формат хранения

- Датасет сохраняется через `dataset.save_to_disk(path)` (HuggingFace `datasets`).
  SFT-сторона читает одной строкой `load_from_disk(path)`.
- Внутри — Arrow/Parquet (колоночный формат, читается стримингом, не грузит всё в RAM).
- **Картинки — встроенные байты** через feature-тип `Image()`, НЕ пути к файлам.
  (Пути = риск «датасет переехал, картинки отвалились» + медленный случайный доступ к диску.)
- Передача — путь на общем диске / в смонтированном томе, а не архив из мелких PNG+JSON.

Пример на стороне data-трека:

```python
from datasets import Dataset, Features, Image, Value, Sequence

features = Features({
    "task_type":   Value("string"),
    "images":      Sequence(Image()),   # встроенные байты
    "current_html": Value("string"),
    "target_html":  Value("string"),
    "instruction":  Value("string"),
})
ds = Dataset.from_list(rows, features=features)
ds.save_to_disk("/data/websight_clean")
```

## 2. Единая схема сэмпла

Одни и те же поля для всех трёх задач. Лишние поля пустуют в зависимости от `task_type`.

| Поле          | Тип              | drafting | polishing | editing |
|---------------|------------------|----------|-----------|---------|
| `task_type`   | string           | `"drafting"` | `"polishing"` | `"editing"` |
| `images`      | list[Image]      | 1 картинка | 2 картинки | 1 картинка |
| `current_html`| string           | — (пусто) | текущий (плохой) HTML | исходный HTML |
| `target_html` | string           | эталонный HTML | эталонный HTML | HTML после правки |
| `instruction` | string           | — (пусто) | — (пусто) | текст правки |

Порядок картинок в `images` фиксирован:
- **drafting:** `[target_screenshot]`
- **polishing:** `[target_screenshot, current_render_screenshot]`
- **editing:** `[screenshot]`

---

## 3. Поля по задачам

### 3.1 Drafting (нужно на MVP)
Картинка → код с нуля.
- `images = [скриншот целевой страницы]`
- `target_html = эталонный HTML`
- `current_html`, `instruction` — пусто.

Это WebSight как есть. Стартуем с него.

### 3.2 Polishing (позже, ждёт рендерер от eval-трека)
Целевая картинка + скриншот текущего рендера + текущий код → исправленный код.
- `images = [target_screenshot, current_render_screenshot]`
- `current_html = текущий (неидеальный) HTML`
- `target_html = эталонный HTML`

Как строить `current_html`/`current_render` (порча эталона или ранние генерации модели) —
зона data-трека, здесь не описывается. SFT нужен только результат в этих полях.

### 3.3 Editing (позже, ждёт данные с инструкциями)
Код/картинка + инструкция на естественном языке → правка.
- `images = [скриншот]`
- `current_html = исходный код`
- `instruction = "сделай кнопку синей" / "добавь футер" / ...`
- `target_html = код после правки`

## 4. Сквозные требования ко ВСЕМ сэмплам

1. **HTML self-contained** — один файл, без внешних зависимостей.
2. **Tailwind предкомпилированный**, не через Play CDN
   (CDN = сетевой вызов + JS-рантайм на каждый рендер → недетерминизм и тормоза у eval).
3. **Серые плейсхолдеры вместо картинок** — каждый реальный `<img>` заменён на серый блок
   фиксированного размера/пропорции. **Одинаковое соглашение везде** (и в `target_html`,
   и в скриншотах, и на eval-стороне), иначе трейн и eval разъедутся.
   Пример: `<div class="bg-gray-300 w-full h-48 rounded"></div>`
4. **Без inline-стилей**, если они запрещены промптом (согласовать единое правило).
5. **Размер скриншотов** — см. §4a ниже (важный пункт, не пропускать).
6. **Гигиена:** дедуп, декотаминация (eval-домены типа Design2Code **никогда** в трейн),
   отсев страниц, не влезающих в бюджет токенов, отсев того, что не рендерится чисто.

---

## 4a. Размер картинок (важно — легко спутать два разных размера)

Есть **два** размера, и путать их нельзя.

**(1) Физический размер скриншота — зона DATA-трека.**
Это размер PNG в пикселях, который вы кладёте в датасет.
- Один и тот же у всех сэмплов.
- Совпадает с вьюпортом eval-рендерера (согласовать с eval-треком).
- Рекомендация: фиксированная ширина (напр. **1280 px**), высота фиксированная
  с обрезкой/паддингом до единого значения (full-page скриншоты переменной высоты
  создают шум при сравнении — не надо).
- **Скриншоты должны быть достаточно крупными, чтобы мелкий текст на кнопках
  и подписях оставался читаемым.** UI текстоплотный — это главная причина.
  Лучше прислать крупнее: SFT всегда может ужать в процессоре, а вот вернуть
  потерянные при сохранении пиксели — нет. Не присылайте 512 px «для лёгкости».

**(2) Разрешение, в котором картинку видит модель — зона SFT-трека.**
Задаётся в коде через `min_pixels`/`max_pixels` процессора Qwen3-VL.
Data-трека не касается, но напрямую зависит от (1): если пришли мелкие скриншоты,
докрутить читаемость текста уже невозможно.

Короткое правило для data-трека: **один фиксированный крупный размер, согласованный
с eval, текст читаем.**

---

## 5. Что нужно когда

- **Сейчас (MVP):** только `task_type == "drafting"`. Даже небольшой очищенный сет
  (сотни–тысячи пар) разблокирует пилотный LoRA-запуск.
- **Позже:** `polishing` — после появления детерминированного рендерера (eval-трек);
  `editing` — после генерации данных с инструкциями (data-трек).

SFT-код на MVP обрабатывает только `drafting`; поля под polishing/editing лежат
готовыми в схеме, обработка добавляется без изменения контракта.

---

## 6. Приёмка (definition of done для передачи)

Датасет считается принятым, если:
- [ ] читается через `load_from_disk(path)` без ошибок;
- [ ] `task_type` проставлен, поля соответствуют таблице из §2;
- [ ] картинки встроены как `Image()` (не пути);
- [ ] все скриншоты одного фиксированного размера, согласованного с eval (§4a), текст читаем;
- [ ] плейсхолдеры согласованы с eval-треком;
- [ ] на 5 случайных сэмплах вручную проверено: HTML self-contained, Tailwind,
      серые плейсхолдеры на месте, нет утечки eval-доменов.

---

## 7. Мелочь под Docker

Путь датасета монтируется в контейнер томом, НЕ кладётся в образ:

```bash
docker run --gpus all -it \
  -v $(pwd):/workspace \
  -v /data/websight_clean:/data \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  <image> bash
```

Тогда SFT-сторона читает `load_from_disk("/data")`.
